# Doc 44 — Synthesised Model Test Plan

**Status**: Active, 18-Apr-26

## Purpose

Systematic calibration of the Bayes compiler across the cartesian product of graph dimensions: context structure, topology, sparsity, epoch structure, and treatment lifecycles. Defines the test infrastructure, the coverage matrix, known model limitations, and the experimental design for characterising recovery degradation.

---

## 1. Architecture

### Truth files

Canonical source: `bayes/truth/` in the dagnet repo. Each `synth-*.truth.yaml` file fully specifies a synthetic graph: topology, edge parameters, context dimensions, sparsity settings, epoch structure, and recovery thresholds. Everything else — graph JSONs, DB rows, param files — is derived by `synth_gen.py`.

A fresh deployment clones dagnet, configures `.private-repos.conf` + DB, and runs the plan runner. Bootstrap handles graph generation and DB population automatically.

### Toolchain

| Tool | Role |
|------|------|
| `bayes/truth/*.truth.yaml` | Canonical graph definitions |
| `bayes/synth_gen.py` | Population simulation, row generation, DB writes |
| `bayes/test_harness.py` | Single-graph MCMC with logging |
| `bayes/param_recovery.py` | Single-graph truth comparison |
| `bayes/run_regression.py` | Parallel orchestrator with multi-layer audit |
| `bayes/regression_plans.py` | Config-driven plan runner with variants |
| `bayes/results_schema.py` | Structured JSON output schema |
| `bayes/plans/*.json` | Named test plans |
| `bayes/plans/generate_sparsity_sweep.py` | Truth YAML generator for sparsity matrix |

### Structured results

Every regression run produces machine-readable JSON at `/tmp/bayes_results-*.json`. Per graph, the JSON includes:

- **quality**: rhat, ess, converged_pct, elapsed_s (all rounded)
- **edges / slices**: per-parameter truth, posterior_mean, posterior_sd, z_score, abs_error, status
- **failures / warnings**: structured dicts with type, edge/slice, param, z_score, threshold, abs_error (not opaque strings)
- **bias**: per-parameter mean_bias, direction, consistency, max_z, max_z_source
- **audit**: DSL, subject counts, binding details (per-edge row counts: raw/regime/final), model flags, LOO scores
- **design**: topology classification, n_edges, n_days, traffic, sparsity parameters, context dimensions with lifecycle windows, epoch structure

The `design` block enables cartesian analysis: group results by `design.topology`, `design.sparsity.frame_drop_rate`, `design.context_dimensions[].lifecycles`, etc.

## 2. Coverage Matrix

### Dimension taxonomy

| Dimension | Levels |
|-----------|--------|
| **Context** | Bare, single-dim MECE, two-dim orthogonal, conditional_p |
| **Topology** | Solo edge, linear chain, branch group (Dirichlet), diamond (join), fan-out, skip, lattice, mirror |
| **Epoch** | Single, mixed (bare + context), staggered (dimension introduction) |
| **Random sparsity** | None, mild (10%/0.01/0.10), moderate (20%/0.03/0.25), severe (35%/0.05/0.40), extreme (50%/0.08/0.50) |
| **Structured lifecycle** | All-throughout, partial overlap (A throughout + B stops 2/3 + C starts 1/3), combined with random |

### Current graph inventory (53 truth files in `bayes/truth/`)

**Existing (29)**: 10 bare topologies, 9 single-dim contexted, 1 two-dim, 2 mixed-epoch, 1 staggered, 2 sparse contexted, plus drift, forecast, and slow-path variants.

**Matrix-filling (6)**: synth-diamond-context-two-dim, synth-diamond-context-mixed, synth-abc-context-two-dim, synth-abc-context-mixed, synth-abc-sparse, synth-context-two-dim-sparse.

**Sparsity sweep (18)**: 3 topologies (solo/chain/diamond) × 6 sparsity configs (4 random levels + lifecycle + lifecycle-sparse). Generated by `bayes/plans/generate_sparsity_sweep.py`.

## 4. Sparsity Calibration

The sparsity sweep answers: how does recovery degrade as a function of data quality, and at what threshold does each topology fail?

### Random degradation axis

Four levels of increasing random data loss, tested on three topologies:

| Level | frame_drop | toggle | initial_absent | traffic | Expected effect |
|-------|-----------|--------|---------------|---------|-----------------|
| sparse-1 | 0.10 | 0.01 | 0.10 | 500 | Minimal — wider posteriors, same recovery |
| sparse-2 | 0.20 | 0.03 | 0.25 | 500 | Moderate — some slice z-scores increase |
| sparse-3 | 0.35 | 0.05 | 0.40 | 200 | Severe — convergence degrades, onset/sigma miss |
| sparse-4 | 0.50 | 0.08 | 0.50 | 100 | Extreme — identifies model failure boundary |

### Structured lifecycle axis

Models treatment switching: baseline A is active throughout, treatment B stops at day 65, treatment C starts at day 33. Controlled by `active_from_day` / `active_to_day` on context dimension values in the truth YAML. The population simulation adjusts weights dynamically — inactive treatments receive no new entrants; row emission is gated per value per day.

Two configs: lifecycle alone, and lifecycle + moderate random sparsity (the hardest real-world case).

### Analysis

The structured JSON output includes `design.sparsity`, `design.topology`, `design.context_dimensions[].lifecycles`, enabling post-hoc cartesian analysis without parsing graph names:

```bash
jq '[.variants.default.graphs[] | {
    topo: .design.topology,
    frame_drop: .design.sparsity.frame_drop_rate,
    traffic: .design.mean_daily_traffic,
    rhat: .quality.rhat,
    ess: .quality.ess,
    max_onset_z: (.bias.onset.max_z // 0),
    passed: .passed
}]' /tmp/bayes_results-plan-sparsity-sweep-full-*.json
```

## 5. Known Model Limitations

### Confirmed and fixed (this session)

1. **Per-slice onset=0 suppression** — `param_recovery.py` suppressed onset recovery output for slices with truth onset=0. Fixed.
2. **Multi-dimension regime selection** — `select_regime_rows` treated orthogonal context dimensions as competing regimes, dropping the second dimension's rows. Fixed: `select_regime_rows_multidim` in `snapshot_regime_selection.py` groups by `context_keys` and runs selection per dimension independently.
3. **Per-slice kappa_lat missing** — batched trajectory path did not create `kappa_lat_slice_vec`. Fixed: BetaBinomial intervals with per-slice kappa_lat in both batched and perslice_traj paths.
4. **Aggregate suppression too aggressive** — `is_exhaustive` used a coverage ratio (>85%) that could silence aggregate evidence from bare-only dates. Fixed: strict emptiness check (`not agg_window and not agg_cohort`).
5. **Upstream latency vars dropped** — per-slice emissions built `_lv` with only the current edge, causing upstream edges to fall back to fixed priors. Fixed: copy `latency_vars` and override only the current edge.
6. **Phase 2 Case A overwrites per-slice p** — `_emit_edge_likelihoods` unconditionally created aggregate `p_cohort_` in Phase 2, discarding per-slice `p_override`. Fixed: guard with `p_override is None`.
7. **Branch group first-sibling template** — Section 6 iterated only the first sibling's slice keys. Fixed: union of all siblings' slice keys, matching Section 2b's pattern.

### Open

8. **`cohort_latency_vars` not per-slice** — on multi-latency paths, cohort slice emissions use aggregate FW-composed latency for upstream edges. Same copy+override pattern as fix 5 but requires tracing the FW composition structure.
9. **`_resolve_path_probability` aggregate-only** — upstream p for per-slice emissions is always aggregate. Cross-edge per-slice p is a Phase C design concern.
10. **Cross-product / dot-product context cells** — model treats dimensions independently (no interaction terms). Per-cell (channel×device) variables not implemented.
11. **`conditional_p` per-slice emission** — conditional_p variables exist at condition level, not per-slice×condition.
12. **Subsumption hierarchy** — not designed.
