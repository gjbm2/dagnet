# Doc 33: CLI Analyse Blockers for Synth Graph Testing

**Date**: 10-Apr-26
**Status**: Both blockers fixed. No remaining gaps — `synth_gen.py --write-files` already writes parameter files with full cohort evidence.
**Context**: Phase B forecast engine testing requires running
`cohort_maturity_v2` analyses on synthetic graphs via the CLI
(`graph-ops/scripts/analyse.sh`). Two structural issues prevent this.

---

## Blocker 1: DSL Subject Doubling

### Symptom

Running:
```
bash graph-ops/scripts/analyse.sh synth-forecast-test \
  "from(ft-hub).to(ft-target).cohort(1-Jan-26:1-Mar-26)" \
  --type cohort_maturity_v2
```

The BE receives:
```
from(ft-hub).to(ft-target).from(ft-hub).to(ft-target).cohort(1-Jan-26:1-Mar-26)
```

Subject resolution finds 0 matches. Analysis returns empty.

### Root Cause

`graph-editor/src/cli/commands/analyse.ts` lines 148-160: when no
explicit `--subject` is provided, the CLI extracts subject clauses
(`from()`, `to()`) from the query DSL via regex and passes them as
`analyticsDsl`:

```typescript
const subjectRe = /\b(from|to|visited|visitedAny|exclude)\([^)]*\)/g;
analyticsDsl = subjectParts.join('.');
```

This produces `analyticsDsl = "from(ft-hub).to(ft-target)"`.

Then `prepareAnalysisComputeInputs` (called at line 181) builds the
per-scenario request using both `analyticsDsl` and `currentDSL`. The
FE's analysis pipeline concatenates the subject back onto the temporal
DSL, producing the doubled form.

### What Should Happen

When the user provides a combined DSL like
`from(x).to(y).cohort(1-Jan-26:1-Mar-26)`, the CLI should either:

- pass it as-is to the BE without re-extracting and re-prepending the
  subject portion, or
- extract subject and temporal parts cleanly and pass them as separate
  fields that `prepareAnalysisComputeInputs` does not re-concatenate

The BE subject resolver expects the full DSL exactly once. The
doubling breaks subject resolution for every multi-hop and single-hop
cohort_maturity_v2 analysis via CLI.

### Affected Paths

- `graph-ops/scripts/analyse.sh` → `src/cli/commands/analyse.ts`
- All CLI-driven analyses with embedded `from()/to()` in the query DSL
- Both v1 (`cohort_maturity`) and v2 (`cohort_maturity_v2`) types
- Also affects `parity-test.ts` when comparing v1 vs v2

---

## Blocker 2: No Stats / Topo Pass in CLI Pipeline

### Symptom

Synthetic graphs generated by `bayes/synth_gen.py` have snapshot data
in the DB and latency params on the graph edges, but no `model_vars`
entries. The `model_vars` array is empty on every edge.

Without `model_vars`, the promoted model fields (`promoted_mu_sd`,
`promoted_sigma_sd`, etc.) are never written. The v2 row builder's
`has_bayes` gate checks `sp.mu_sd > 0`, which reads from
`edge_params.bayes_mu_sd`, which reads from `promoted_mu_sd` — all
zero. The MC block is skipped entirely. No fan chart.

### Root Cause

In the FE, `model_vars` are populated by the analytic topo pass
(`localAnalysisComputeService.ts` → `beTopoPassService.ts`), which
runs automatically when a graph is opened in the UI. The topo pass:

1. Fits latency distributions from snapshot evidence
2. Writes `model_vars` entries (analytic, analytic_be) with mu, sigma,
   onset, mu_sd, sigma_sd, etc.
3. Runs `applyPromotion` which writes `promoted_mu_sd` etc. to the
   edge's latency block

The CLI has no equivalent. `analyse.sh` loads the graph from disk,
sends it to the BE, and expects the graph to already contain
everything needed for analysis. A freshly generated synth graph has
the ground-truth latency params on the edges but no model_vars —
because no topo pass has ever run on it.

### What Should Happen

The CLI pipeline needs a way to populate `model_vars` before running
analyses. Options:

1. **CLI topo pass**: add a `--run-topo-pass` flag (or make it
   automatic) that calls the BE's `/api/lag/topo-pass` endpoint
   before running the analysis. This mirrors what the FE does on graph
   open.

2. **Synth generator writes model_vars**: `synth_gen.py` already
   writes latency params from truth. It could also write synthetic
   `model_vars` entries with the truth params and reasonable SDs.
   This avoids depending on the topo pass at all.

3. **BE reads edge latency directly**: the v2 row builder could read
   `mu_sd` from the edge's `p.latency` block (where the synth
   generator writes it) rather than requiring promoted fields from
   `model_vars`. This is what the current code attempts with the
   `promoted_mu_sd` fallback, but it fails because the synth
   generator doesn't write `promoted_mu_sd`.

Option 1 is the most correct — it exercises the real pipeline. Option
2 is the quickest to implement. Option 3 is a partial workaround.

### Affected Paths

- `bayes/synth_gen.py` → writes graph JSON without model_vars
- `graph-editor/lib/runner/span_adapter.py` → reads `promoted_mu_sd`
- `graph-editor/lib/runner/span_kernel.py` → reads `promoted_mu_sd`
  in `mc_span_cdfs`
- `graph-editor/lib/runner/cohort_forecast_v2.py` → `has_bayes` gate
- `graph-editor/lib/api_handlers.py` → `_read_edge_model_params`
  reads `bayes_mu_sd` from model_vars fallback

---

## Impact

These two blockers prevent all CLI-based testing of the v2 forecast
engine on synthetic graphs. Without CLI testing:

- The δ₀ non-latency hop fix cannot be verified
- Parity tests between v1 and v2 cannot run
- The three-tier carrier hierarchy cannot be validated against known
  ground truth
- E_i frontier exposure behaviour cannot be checked on partially
  mature spans

Both blockers must be resolved before Phase B acceptance gates can be
evaluated.

## Resolution (10-Apr-26)

### Blocker 1 Fix: DSL Subject/Temporal Splitting

The CLI now splits a combined DSL (e.g.
`from(x).to(y).cohort(1-Jan-26:1-Mar-26)`) into subject and temporal
parts **before** passing them to `prepareAnalysisComputeInputs`. A
`splitDsl()` helper in `analyse.ts` extracts subject clauses
(`from/to/visited/visitedAny/exclude`) and strips them from the
temporal remainder. When `--subject` is explicit, the query DSL is
left untouched. This prevents the BE from seeing subject clauses
twice (once in `analytics_dsl`, once in `effective_query_dsl`).

### Blocker 2 Fix: `--topo-pass` Flag

A `--topo-pass` flag added to `analyse.ts`. When set, the CLI:

1. Builds cohort data from `bundle.parameters` (parameter files
   loaded from disk by `diskLoader`), converting per-day parallel
   arrays (`n_daily`, `k_daily`, `dates`, `median_lag_days`, etc.)
   into per-date `CohortData` records keyed by edge UUID
2. POSTs the graph + `cohort_data` to `/api/lag/topo-pass`
3. Writes returned stats as `promoted_*` fields on the base graph
   edges before dispatching the analysis

This is necessary because the CLI's aggregate step (`fetchItems` in
`from-file` mode) calls `getParameterFromFile` which hits
`fileRegistry.restoreFile()` → IDB — and IDB is unavailable in Node.
The FE topo pass (Stage 2 of the fetch pipeline) never fires in CLI
context, so `model_vars` and `promoted_*` fields are never populated
via the normal path. The `--topo-pass` flag bypasses IDB entirely by
reading cohort evidence directly from disk-loaded parameter files.

### Additional Fixes Discovered During E2E Validation

**BE snapshot cache bypass** (`--no-snapshot-cache`): the BE's
snapshot service caches query results in memory. After `synth_gen.py`
repopulates the DB, stale cached empty results persist. The new flag
sets `globalThis.__dagnetComputeNoCache`, causing `graphComputeClient`
to append `?no-cache=1` to BE requests. The BE middleware sets a
per-thread cache bypass flag.

**Cohort date parsing fix** (`analysis_subject_resolution.py`):
`_extract_time_bounds` parsed `cohort(anchor,start:end)` incorrectly
— the regex captured `anchor,start-date` as the start date, which
failed date parsing and fell through to today's date. This caused
`anchor_from` to be set to today, filtering out all historical
snapshot data. Fixed by making the anchor prefix optional in the
regex: `cohort\((?:[^,)]*,)?([^:,]*):([^)]*)\)`.

### Verified E2E Workflow

```
# 1. Generate synth data (param files + snapshot DB rows)
python synth_gen.py --graph synth-simple-abc --write-files

# 2. Run v2 analysis with topo pass and cache bypass
bash graph-ops/scripts/analyse.sh synth-simple-abc \
  "from(simple-a).to(simple-c).cohort(simple-a,12-Dec-25:21-Mar-26)" \
  --type cohort_maturity_v2 --topo-pass --no-snapshot-cache
```

Result: 120 data rows, 119 with rates, 99 with fan bands, promoted
fields (`promoted_mu_sd=0.02`) flowing through to the BE correctly.
